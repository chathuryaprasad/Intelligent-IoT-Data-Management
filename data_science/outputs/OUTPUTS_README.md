# Outputs Folder

This folder stores benchmark outputs generated by the Models team anomaly detection pipeline.

The files in this folder are produced automatically by commands such as:

```bash
python pipeline.py datasets/complex.csv --benchmark
python pipeline.py datasets/complex_clean.csv --benchmark --train-test
python run_all_benchmarks.py
```

Most files inside `outputs/` are generated evidence files. They should usually not be manually edited.

---

## Purpose

The `outputs/` folder helps the team:

- store per-benchmark results
- compare detector performance
- inspect generated plots
- prepare handover documentation
- support project updates and presentations
- provide Backend and Frontend with examples of model output structure

The outputs are useful for analysis and documentation, but they are not the core source code of the pipeline.

## Benchmark Output Folders

## `synthetic_benchmark/`

Generated by:

```bash
python pipeline.py datasets/complex.csv --benchmark
```

or as part of:

```bash
python run_all_benchmarks.py
```

Purpose:

- evaluates detectors on `datasets/complex.csv`
- injects synthetic anomalies using `anomaly_injector.py`
- compares detector predictions against exact injected labels
- works as a controlled sanity check for detector behaviour

Use this folder when checking whether detectors work correctly on known artificial anomalies.

---

## `train_test_benchmark/`

Generated by:

```bash
python pipeline.py datasets/complex_clean.csv --benchmark --train-test
```

or as part of:

```bash
python run_all_benchmarks.py
```

Purpose:

- splits the dataset into train and test sections
- fits trainable detectors on the training section where supported
- injects synthetic anomalies only into the test section
- evaluates detectors on unseen test data

Use this folder when checking whether detectors generalise better when fitting and evaluation are separated.

Note:

The current `run_all_benchmarks.py` may use `datasets/complex.csv` for train/test depending on the latest branch state. If the team wants the cleaner controlled train/test setup, use `datasets/complex_clean.csv` directly or update the runner.

---

## `nab_benchmark/`

Generated by:

```bash
python pipeline.py "https://raw.githubusercontent.com/numenta/NAB/master/data/realKnownCause/machine_temperature_system_failure.csv" --benchmark --label-source nab --nab-label-file "https://raw.githubusercontent.com/numenta/NAB/master/labels/combined_windows.json" --nab-dataset-key "realKnownCause/machine_temperature_system_failure.csv"
```

or as part of:

```bash
python run_all_benchmarks.py
```

Purpose:

- evaluates detectors against real labelled anomaly windows from the Numenta Anomaly Benchmark
- uses `nab_loader.py` to convert NAB windows into boolean labels
- provides the most realistic benchmark among the current modes

Important limitation:

The project currently uses pointwise precision, recall, F1, AUC-ROC, FPR, and FNR for NAB. It does not implement the official NAB scoring profiles yet.

## Per-Benchmark Output Files

Each benchmark mode can generate the following files.

## `benchmark_results.csv`

This is the main numeric results table.

It usually contains:

```text
detector
precision
recall
f1
auc_roc
n_predicted
n_actual
true_positives
false_positives
true_negatives
false_negatives
false_positive_rate
false_negative_rate
model
```

Use this file for:

- exact metric comparison
- report tables
- spreadsheet analysis
- checking detector trade-offs
- possible backend/frontend integration examples

Example interpretation:

- high precision means fewer false alarms
- high recall means fewer missed anomalies
- high F1 means better balance between precision and recall
- high AUC-ROC means stronger score separation between normal and anomalous points

---

## `benchmark_results.json`

This contains the same benchmark metrics as the CSV file, but in JSON format.

Use this file for:

- programmatic access
- frontend/backend experiments
- structured reporting
- checking how benchmark outputs could be consumed by another service

---

## `metrics_comparison.png`

This plot compares detector metrics such as:

```text
precision
recall
f1
auc_roc
```

Use this plot to quickly see which detectors perform best in a specific benchmark mode.

Good for:

- presentation slides
- handover summaries
- mentor updates
- model comparison discussion

---

## `runtime_comparison.png`

This plot compares detector runtime in seconds.

Use this plot to understand:

- which detectors are fastest
- which detectors may be expensive in real-time use
- whether a detector is practical for backend or dashboard integration

Important:

Only detectors that return a valid `runtime` value appear correctly in this plot.

---

## `confusion_summary.png`

This plot shows true positives, false positives, false negatives, and true negatives for each detector.

Use this plot to understand:

- which detectors produce many false alarms
- which detectors miss many anomalies
- whether a detector is conservative or sensitive
- whether a high F1 score hides a specific weakness

Definitions:

```text
TP = anomaly correctly flagged
FP = normal point incorrectly flagged
TN = normal point correctly ignored
FN = anomaly missed
```

---

## `model_flags_timeline.png`

This plot shows when each detector flagged anomalies along the time axis.

It usually includes a ground-truth row when labels are available.

Use this plot to understand:

- whether detectors flag the same anomaly periods
- whether a detector fires too often
- whether a detector misses full anomaly windows
- whether predictions align with injected or labelled anomaly periods

This is especially useful for time-series interpretation because numeric metrics alone do not show when detections occurred.

---

## `benchmark_report_summary.txt`

This is a short generated text summary for one benchmark mode.

It usually identifies:

```text
Best F1
Best AUC-ROC
Most conservative detector
Most sensitive detector
Fastest detector
```

Use this file for:

- quick standup updates
- PR descriptions
- high-level model comparison
- quick explanation to non-technical teammates

---

## `roc_curves.png`

This plot shows ROC curves for detectors that produce continuous anomaly scores.

Use this plot for threshold-independent comparison.

Important:

- detectors without a `score` output may not appear
- ROC may be skipped if the labels contain only one class
- AUC values are more meaningful when score direction is consistent, where higher score means more anomalous

---

## Final Cross-Benchmark Outputs

These files are generated after running multiple benchmark modes, usually through:

```bash
python run_all_benchmarks.py
```

## `final_benchmark_summary.csv`

This file combines detector results across all available benchmark modes.

It should include a `benchmark_type` column, for example:

```text
synthetic
train_test
nab
```

Use this file to compare the same detector across different benchmark types.

Useful questions:

- Is a detector consistently strong?
- Does a detector only work well on synthetic data?
- Which detector performs best on real NAB data?
- Which detectors generalise better in train/test mode?

---

## `final_benchmark_report.txt`

This is the final high-level benchmark report.

It summarises each benchmark mode and identifies key winners such as:

```text
Best F1
Best AUC-ROC
Fastest detector
Most conservative detector
Most sensitive detector
```

Use this file for:

- final handover documentation
- mentor updates
- presentations
- model selection discussions

---

## `final_f1_comparison.png`

This plot compares F1 score across benchmark types.

Use this to see which detectors maintain a good precision/recall balance across different evaluation modes.

---

## `final_auc_comparison.png`

This plot compares AUC-ROC across benchmark types.

Use this to compare detector score quality across benchmark modes.

Important:

Some detectors may have missing AUC values if they do not return continuous scores or if ROC-AUC cannot be computed.

## How to Regenerate Outputs

From inside `data_science/`:

```bash
python run_all_benchmarks.py
```

For synthetic only:

```bash
python run_all_benchmarks.py --skip-train-test --skip-nab
```

For train/test only:

```bash
python run_all_benchmarks.py --skip-synthetic --skip-nab
```

For NAB only:

```bash
python run_all_benchmarks.py --skip-synthetic --skip-train-test
```

Or run individual pipeline commands:

```bash
python pipeline.py datasets/complex.csv --benchmark
python pipeline.py datasets/complex_clean.csv --benchmark --train-test
python pipeline.py "https://raw.githubusercontent.com/numenta/NAB/master/data/realKnownCause/machine_temperature_system_failure.csv" --benchmark --label-source nab --nab-label-file "https://raw.githubusercontent.com/numenta/NAB/master/labels/combined_windows.json" --nab-dataset-key "realKnownCause/machine_temperature_system_failure.csv"
```

## Should These Outputs Be Committed?

Usually, generated outputs should not be committed.

Recommended rule:

```text
Commit source code and README documentation.
Avoid committing generated CSV, JSON, TXT, and PNG outputs unless they are final evidence for handover.
```

Reason:

- benchmark outputs can change after every run
- synthetic injection may depend on random seeds
- generated PNG/CSV/JSON files can clutter Git history
- the source code should be the main source of truth

However, the team may intentionally commit a final set of outputs if they are being used as final evidence for assessment, handover, or presentation.

If outputs are ignored, keep the folder with:

```text
outputs/.gitkeep
```

## Current Known Benchmark Takeaways

Based on the latest benchmark evidence used for handover:

### Synthetic benchmark

- `LOFDetector` achieved the best F1.
- `ThresholdADDetector` achieved the best AUC-ROC.
- `LOFDetector` was the most sensitive by recall.
- `COPODDetector` and `ThresholdADDetector` were among the fastest detectors.

### Train/test benchmark

- `OCSVMDetector` achieved the best F1.
- `OCSVMDetector` achieved the best AUC-ROC.
- `OCSVMDetector` was the most sensitive by recall.
- `COPODDetector` and `ThresholdADDetector` were among the fastest detectors.

### NAB benchmark

- `InterQuartileRangeAD` achieved the best F1.
- `ThresholdADDetector` achieved the best AUC-ROC.
- `InterQuartileRangeAD` was the most sensitive by recall.
- `ThresholdADDetector` and `COPODDetector` were among the fastest detectors.

Overall:

- `OCSVMDetector` performed strongly in controlled benchmark modes, especially in train/test evaluation.
- `LOFDetector` achieved the strongest synthetic F1 score after being included in the benchmark comparison.
- `ThresholdADDetector` showed very strong AUC-ROC and precision, but its recall was lower, meaning it behaved as a conservative detector.
- `COPODDetector` and `InterQuartileRangeAD` performed strongly on NAB-style real anomaly windows.
- NAB results should be interpreted carefully because the current evaluation is pointwise, not official NAB scoring.
- No detector is best across every benchmark mode.

## Common Issues

### `outputs/` folder does not appear

Make sure the benchmark command completed successfully.

Also make sure you are running commands from inside:

```text
data_science/
```

### NAB output is missing

Possible causes:

- no internet connection
- NAB raw GitHub URL unavailable
- incorrect `--nab-dataset-key`
- missing `--nab-label-file`
- command stopped before report generation

### ROC curve is missing

Possible causes:

- detector output does not include `score`
- labels only contain one class
- ROC plotting was not triggered in that benchmark path

### Runtime comparison is incomplete

Some detectors may not return `runtime`.

Only detectors with valid runtime values are included in the runtime plot.

### Final summary is missing

Run:

```bash
python run_all_benchmarks.py
```

The individual `pipeline.py` benchmark command creates per-mode outputs, but the final cross-benchmark files are created by the full benchmark runner and final report step.
